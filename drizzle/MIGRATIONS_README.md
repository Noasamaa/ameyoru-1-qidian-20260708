# Migrations: drift & remediation notes

This app was migrated SQLite → MySQL 8 (Drizzle ORM). The versioned `.sql`
migrations under `drizzle/` have drifted from both `src/db/schema.ts` and the
live MySQL schema. This note records exactly what is inconsistent and how to
properly fix it. **Do not hand-edit `meta/_journal.json` or the `meta/*_snapshot.json`
files** — those are machine-generated by drizzle-kit and editing them by hand
risks corrupting the migration metadata further.

## (a) Journal / snapshot inconsistencies

Observed in `meta/_journal.json` and `meta/`:

1. **0002 tag mismatch.** `_journal.json` entry `idx: 2` has
   `"tag": "0002_boring_blackheart"`, but the actual file on disk is
   `0002_gift_records.sql`. The journal tag and the filename disagree.
2. **Missing 0002 snapshot.** `meta/` contains `0000_snapshot.json`,
   `0001_snapshot.json`, and `0003_snapshot.json` — there is **no
   `0002_snapshot.json`**. The 0002 step was applied without a snapshot being
   committed.
3. **0003 prevId points at 0001 (chain skips 0002).** Snapshot `id`/`prevId`
   continuity is:
   - `0000_snapshot.json` id `e417d383…`  prevId `00000000…`
   - `0001_snapshot.json` id `b835d7c7…`  prevId `e417d383…`
   - `0003_snapshot.json` id `08b18a64…`  prevId `b835d7c7…` ← **points back at
     0001's id**, because 0002 never produced a snapshot. The lineage therefore
     jumps 0001 → 0003 and 0002 is orphaned from the snapshot chain.

## (b) 0003 is a no-op; gift-payment / deposit columns came from manual ALTERs

- `0003_long_sharon_ventura.sql` is literally `SELECT 1;` — a no-op marker. It
  applies no DDL.
- Yet `0003_snapshot.json` (and the live DB) contain columns that **no `.sql`
  file ever creates**:
  - `gift_record.submitter_id` `varchar(64)` NOT NULL (no default)
  - `gift_record.settle_status` `enum('UNSETTLED','SETTLED')` NOT NULL DEFAULT `'UNSETTLED'`
  - `gift_record.settled_at` `datetime(3)` NULL
  - `gift_record.paid_method` `enum('WECHAT','ALIPAY')` NULL
  - `user.deposit_paid` `boolean` NOT NULL DEFAULT `false`
  - indexes `gift_record_settle_idx` (`settle_status`,`created_at`) and
    `gift_record_sender_idx` (`sender_nickname`)

  `0002_gift_records.sql` creates `gift_record` **without** the four payment
  columns and with only the `player_idx` / `created_idx` indexes, and never adds
  `user.deposit_paid`. These were added to the live DB via **manual `ALTER`
  statements run outside drizzle**, then folded into `0003_snapshot.json`. As a
  result, applying `0000`→`0003` to a fresh database produces a schema that is
  **missing** all of the above.

## Interim stopgap: `0004_lonely_dazzler.sql`

Because drizzle-kit is not runnable in this environment (no `node_modules`),
`0004_lonely_dazzler.sql` is a **hand-authored** forward migration that emits the
DDL missing from `.sql` history so a fresh `0000`→`0004` run matches
`src/db/schema.ts`. It also adds the `order_status_start_idx` composite index
(M8) and a defense-in-depth unique on `customer_balance_txn_player`
(`txn_id`,`player_id`).

- Column types/null/default/enum in `0004` were copied verbatim from
  `schema.ts` and cross-checked against `0003_snapshot.json`, so drizzle-kit
  should report **zero drift** for those columns.
- **`0004` is an interim stopgap, not the canonical fix.** It was written by
  hand and has **no accompanying snapshot/journal entry** (we did not hand-edit
  the meta). MySQL 8 `ALTER TABLE … ADD COLUMN` does not reliably support
  `IF NOT EXISTS`, so `0004` is idempotency-by-convention: **apply once, and on
  the live DB (which already has these columns) skip it entirely.**

## (c) Canonical remediation — requires tooling not available here

The proper fix needs drizzle-kit, which needs dependencies installed:

```bash
npm install
```

Then either:

- **Regenerate from schema (preferred to keep history):**
  ```bash
  npx drizzle-kit generate
  ```
  drizzle-kit diffs `src/db/schema.ts` against the latest snapshot and emits a
  real migration **plus** its matching `*_snapshot.json` and `_journal.json`
  entry. Review the generated SQL; if it reproduces what `0004` does by hand,
  prefer the generated pair and retire the hand-authored `0004` + this stopgap.

- **Or rebuild a clean baseline from the live DB:**
  ```bash
  npx drizzle-kit introspect
  ```
  This reverse-engineers a fresh baseline migration + snapshot from the actual
  live MySQL schema, sidestepping the drifted history entirely.

After either, **verify `prevId` continuity**: each `meta/NNNN_snapshot.json`
`prevId` must equal the previous snapshot's `id`, the `_journal.json` `tag`s must
match the on-disk `.sql` filenames, and there must be a snapshot for every
journal entry. Fix the 0002 tag mismatch / missing-0002-snapshot / 0003-prevId
gap **only via regenerated metadata**, never by hand.
